## §1 Architecture summary of changes

### Schema deltas (final v0.3 row shape additions)

```ts
// Added to MemoryRow at src/schema.ts:84-108
agent_id:       string;   // backfill "legacy-claude-local"; new: sha256(ed25519_pub) 64-hex
agent_alias:    string;   // backfill "claude"; e.g. "claude"|"codex"|"human:hammad"
capabilities:   string;   // JSON-serialized string[], default "[]"
valid_from:     number;   // unix ms, default = created_at
valid_to:       number;   // unix ms, 0 = currently valid
review_after:   number;   // unix ms, 0 = no review scheduled
```

Existing columns unchanged. `visibility` (already in schema at `schema.ts:94`) is preserved but now **enforced** at every read path.

### Tool surface deltas

- All 10 existing tools accept a caller identity injected by the transport layer (HTTP bearer → caller_agent_id). No new MCP tool method signatures.
- `memory_health` extended with the telemetry contract: 12 Codex metrics + 3 cross-check gauges = 13 health fields total (§4).
- New tool: `memory_review_stale` — returns rows where `review_after > 0 AND review_after <= now`.

### Read-path filter (canonical form for v0.3)

Every read path (semantic, structured, hybrid search; loadSession's 5 tiers; task_list) must apply:

```sql
status = 'active'
AND (visibility = 'shared' OR agent_id = '{caller_agent_id}')
AND (valid_to = 0 OR valid_to >= {now})
AND (scope = 'global' OR scope = '{requested_scope}')
```

**Critical:** scope is not identity, and validity is not visibility. The three filters compose independently. v3 §1 Q3 makes this explicit, repeated here so implementers don't conflate them.

### Phase 3 — O2 refined: validity + review semantics

**Boundary:** Phase 3 adds temporal correctness without touching dedup. Complete when every row has validity/review fields, lifecycle respects `valid_to`, stale-review rows discoverable without expiring durable memories. Phase 4 unlockable because dedup restructure can preserve the final v0.3 row shape.

**Files in src/ change:**
- `schema.ts`: add `valid_from`, `valid_to`, `review_after` columns + Zod schemas
- `store.ts`: set defaults in `createNew()` [`code: src/store.ts:237-269`]; preserve fields in `update()`, `normalizeRow()`
- `lifecycle.ts`: treat `valid_to < now` per policy; `review_after` is surfacing only [`code: src/lifecycle.ts:62-70`]
- `index.ts` + `http-server.ts`: add `memory_review_stale` tool
- `pmm-bridge.ts`: map defaults for PMM rows missing these fields

**Schema delta:**

| Column | Type | Default | Constraint |
|---|---|---|---|
| `valid_from` | number (unix ms) | `created_at` | finite; `valid_from <= valid_to` when `valid_to != 0` |
| `valid_to` | number (unix ms) | `0` (sentinel = currently valid) | `0` or `>= valid_from` |
| `review_after` | number (unix ms) | `0` (no review scheduled) | non-negative finite. `0` = no review. Non-zero values MAY be in the past — that's the signal `memory_review_stale` looks for. Write-time validation only checks finiteness and non-negativity, not future-ness. |

**Per-type backfill model** (from v3 §1 Q2, refined by Codex's `review_after` primitive):

| Type | `valid_from` | `valid_to` | `review_after` |
|---|---|---|---|
| `context` | `created_at` | `expires_at` if > 0, else `0` | `0` (TTL handles it) |
| `blocker` | `created_at` | `expires_at` if > 0, else `0` | `0` |
| `task` | `created_at` | `0` | `created_at + 7d` (stale-review) |
| `observation` | `created_at` | `0` | `created_at + 60d` (or earlier if not cited by decision/lesson) |
| `fact` | `created_at` | `0` (drift-prone fields get explicit `valid_to`) | application-set |
| `decision` | `created_at` | `0` | `created_at + 120d` |
| `preference` | `created_at` | `0` | `created_at + 180d` |
| `lesson` | `created_at` | `0` | `created_at + 270d` |
| `procedure` | `created_at` | `0` (version-bound, not time-bound) | `created_at + 120d` |
| `identity` | `created_at` | `0` | `0` (periodic review only, not column-driven) |
| `relationship` | `created_at` | `0` | `created_at + 270d` |

**Read-path filter additions:**

Normal reads exclude invalid rows:
```sql
AND (valid_to = 0 OR valid_to >= {now})
```

`memory_review_stale` returns:
```sql
status = 'active' AND review_after > 0 AND review_after <= {now}
```

**Migration step:**
- Snapshot: take fresh snapshot before O2 (don't rely on Phase 0)
- Backfill rule: per the table above
- Dry-run validation: spot-check 10 rows per type (the migration invariant from v3 §3); confirm no active row disappears from `memory_load_session` unless `valid_to` is already past
- Acceptance gate: row count unchanged; no row with `valid_to < valid_from` (except `valid_to = 0`); `memory_review_stale` returns only rows with `review_after <= now`; existing `expires_at` semantics for context/blocker unchanged

**Rollback path:** prefer snapshot restore. Code-only rollback possible if new columns are ignored, but verify session-load tier order matches Phase 0 baseline before declaring rollback complete.

**Feature flag:** `SUPERCLAUDE_TEMPORAL_FILTERS=report|enforce` — default `report` for O2 dry-run, `enforce` before release.

**Acceptance test:**
```powershell

```
Assertions:

- `review_after` does NOT archive rows
- `valid_to` filters normal search/session results

## §3 Gotchas v3 missed (from Codex consult)

Five gotchas the v3 backlog left implicit. Each addressed in the phase plan above.

1. **Visibility cannot be enforced without caller identity.** v3 listed O1 before O7, but current HTTP has no auth [`code: src/http-server.ts:343-421`] and current MCP tools pass only user filters to store [`code: src/index.ts:91-110`]. O1 and O11 need a minimal auth slice EARLIER than v3 implied. **→ Fixed in Phase 1 (batched).**

2. **`memory_load_session` is its own leakage path.** It does direct table queries for 5 tiers [`code: src/store.ts:425-455`], not a shared search helper. Visibility/validity filters must be added to EACH tier, plus PMM bridge session loading [`code: src/pmm-bridge.ts:148-196`]. **→ Phase 1 explicitly enumerates all 5 tiers in the file-changes list.**

3. **`notify-handler` already receives `agent_id` but discards it when saving.** Inbound schema has `agent_id` [`code: src/notify-handler.ts:22-31`] but `store.save()` gets only content/type/scope/priority/topics/source/visibility/task_data [`code: src/notify-handler.ts:118-127`]. **If not fixed, bridge-origin memories will be falsely attributed to the local default agent.** → Phase 1 explicitly fixes notify-handler.

4. **Vector-shape mismatches can silently evade migration tests.** Health counts invalid dimensions [`code: src/store.ts:546-580`] and dedup skips mismatched candidates [`code: src/store.ts:749-752`]. Backfill must NOT rewrite or drop rows for malformed vectors — it should count them, preserve content, let telemetry alert. **→ Phase 1 backfill preserves mismatched-vector rows (no rewrite, no drop) — assert this by adding `mismatch_count_unchanged_post_backfill` to the Phase 1 acceptance test (counted via `memory_health.vector_dimensions.invalid_rows` pre vs post). Phase 2 adds `vector_shape_mismatch_total` as a CRITICAL alert metric. Phase 4 dedup explicitly skips mismatched candidates and the acceptance test asserts this is visible (counted, not silent).**

5. **LanceDB schema evolution is not SQL `ALTER TABLE`.** Current table is created by adding a seed row and deleting it [`code: src/store.ts:156-158`; `src/schema.ts:112-137`]. Adding columns probably means rewriting/upserting rows or table recreation. **Rollback should be snapshot-based, not "drop column".** → Phase 1 rollback path explicitly says "restore snapshot, not partial column deletion".

## §4 Codex's risk pushback (accepted, applied)

> *"I would push back on v3's order putting O7 HTTP auth as P1 after O1/O11. O1 visibility enforcement is not meaningful until the server can identify the caller. ... Move a minimal O7 slice into Phase 1: bearer token from env, caller identity resolution, and 401 on missing/invalid token for `/mcp` and `/notify`."*

**Accepted.** v3 had O7 at P1 (separate from P0 O1/O11). This spec moves O7 into Phase 1 as part of the auth+identity batch. The reasoning:
- `agent_id` column without caller identity = column you can't populate truthfully
- `visibility: private` row without auth check = column you can't enforce
- All three (O11 column, O1 enforcement, O7 auth) form one indivisible security model

The v3 priority backlog is updated implicitly: P0 now contains O1+O7+O11+migration playbook+telemetry (= Phase 0 + Phase 1 + Phase 2). P1 contains O2+O9 (= Phase 3 + Phase 4).
