# AMS v1.1 Migration And Monitor Plan

Date: 2026-05-27

Status: implementation-approved

## Goal

Turn AMS v1 from a usable local memory CLI into an operated memory system:

```text
curated migration -> reviewable ledger -> apply only safe items -> monitor health -> dashboard/status
```

## Answer To Dashboard/Log Question

AMS v1 already stores records:

- `cem.sqlite` for traces, atoms, cards, validations, and decisions;
- `traces.jsonl` for raw trace intake;
- `directives.json` for pinned directives.

AMS v1.1 adds operator records:

- `migration-runs.jsonl`;
- `migration-latest.json`;
- `migration-latest.md`;
- `monitor-runs.jsonl`;
- `monitor-latest.json`;
- `monitor-latest.md`;
- `ams dashboard` for a current summary.

## Migration Rule

Do not bulk-import Codex memory.

The migrator only reads the Codex memory registry and only extracts the `C:\Dev\Builds\Agentic Memory System` task group. Entries are classified as:

- `pin`: explicit directives and repo guardrails;
- `remember`: verified operational experience with action/outcome shape;
- `skip`: stale snapshots, unrelated project memory, vague summaries, and anything that would pollute CEM.

Dry-run is the default operating posture. Apply only uses the curated plan.

## Monitor-0 Checks

Monitor-0 checks:

- AMS root exists;
- SQLite store exists;
- directive file exists;
- minimum directive count is present;
- at least one learned card exists;
- action brief includes Waki boundary;
- action brief includes pytest/synthetic verification rule;
- action brief includes TODO continuation rule;
- optional deep eval still reports CEM write-path metrics.

## Commands

```powershell
python scripts/ams.py migrate dry-run
python scripts/ams.py migrate apply
python scripts/ams.py monitor
python scripts/ams.py monitor --deep
python scripts/ams.py dashboard
```

## Acceptance Tests

- migration dry-run writes latest JSON/Markdown and does not mutate memory;
- migration apply pins directives idempotently and does not duplicate learned cards;
- monitor writes JSONL plus latest JSON/Markdown;
- dashboard reads latest migration/monitor records;
- monitor fails visibly when required brief directives are missing;
- all commands support JSON output.

## Non-Scope

- no web dashboard;
- no always-on background process yet;
- no cron/heartbeat automation until Monitor-0 exists and passes;
- no unrelated project migration;
- no Waki workspace reads or writes.
