## §1 Problem formulation — agent memory as a sequential decision problem

> The MIT-12 protocol shape applies here.

**SOURCE FACT** [`mit12: book-04-chunk-0137 pp.573-577`]: *Algorithms for Decision Making* describes memory-bounded heuristic search for Dec-POMDPs as keeping "only κ_max conditional plans per agent" — a deliberate bound on retained policies, traded against tractability. The decision-theoretic framing: a joint policy π = ⟨π₁...πₙ⟩ over partially-observed shared state, where the utility U(b₀, π) is maximized subject to a depth-d and κ_max plan budget.

**SOURCE FACT** [`mit12: book-03-chunk-0313 pp.952-954`]: *Machine Learning Systems* on memory hierarchy: "The primary goal of memory allocation is to minimize latency and reduce power consumption by keeping frequently accessed data as close as possible to the processing elements." The placement-allocation trade-off space (book-03-chunk-0314): granularity, spatial-vs-temporal, locality, communication, dataflow ordering — each with reciprocal costs.

**INFERENCE**: Agent memory is a 4-tuple ⟨S, W, R, L⟩ — state space, write semantics, read semantics, lifecycle policy — being optimized as a *bounded* sequential decision problem. "Bounded" is doing work: every system in §4 is choosing a κ_max-shaped tradeoff between *retention fidelity* (keep more) and *retrieval tractability* (keep less). The Dec-POMDP frame is exact: each MCP tool call is a noisy observation of the user's state; the memory's read function selects context, which biases the agent's next action, which produces the next observation. This is not a vague analogy — it is the same structure.

**JUDGMENT for SuperClaude**: SC's `findCandidates LIMIT 100` [`code: src/store.ts:735`] is its κ_max. The bound is principled (not arbitrary): finite per-(type, scope) candidate set, full O(N·d) cosine inside. The architectural question is not "is the bound right?" but "is the κ_max-selection *policy* right?" Currently SC's selection is `ORDER BY arbitrary LIMIT 100` — closer to random sampling than to value-of-information selection. A value-of-information policy (preferring candidates whose semantic neighborhood is densest, or whose access_count is highest, or whose age is most recent) would dominate uniformly within the same compute budget.

**JUDGMENT for Codex memory (current sub-project C)**: Inherit the κ_max design — it's textbook. But pick the selection policy deliberately rather than by default.

**OPEN QUESTION** that MIT-12 cannot answer: what is the *empirical* κ_max breakpoint at which SC's dedup quality degrades visibly in Hammad's actual usage distribution? Requires telemetry on the live store, not theory.

## §2 SuperClaude architecture — source-grounded (descriptive, not MIT-12 territory)

All `[verified]` from `C:\Users\7amma\.claude\mcp-servers\superclaude-memory\src\`. Per protocol, no SOURCE FACT/INFERENCE/JUDGMENT/OPEN QUESTION shape — these are repo facts, not design claims.

### 2.1 Storage substrate

- **Engine:** LanceDB embedded, single table `agent_memories` at `~/.claude/memory/lancedb` [`code: src/store.ts:119, 145-159`].
- **Init:** lazy on first tool call, avoiding MCP startup timeout [`code: src/store.ts:131-160`].
- **Row schema:** 21 columns including `id, content, vector (1024-d), type, scope, priority, topics, status, visibility, created_at, updated_at, accessed_at, access_count, expires_at, supersedes, superseded_by, source_session, source_type, media_*, task_data` [`code: src/schema.ts:84-108`].

### 2.2 Embedding

- BGE-M3, 1024 dims, via `@openclaw/local-embeddings` [`code: src/schema.ts:114`].
- Asymmetric: `embedText(text, "passage")` for writes, `"query"` for reads [`code: src/store.ts:170, 281`].

### 2.3 Type system

11 types × 4 priorities × 3 statuses × 7 source types × 5 task statuses [`code: src/schema.ts:5-58`]. The most opinionated piece of the design: pre-commits to a taxonomy that pays off in `loadSession` and `taskList`.

### 2.4 Search modes

| Mode | Mechanism | Complexity |
|---|---|---|
| `semantic` | `vectorSearch(qV).limit(k×3).filter(status='active')` + post-filter | **O(log N)** ANN + **O(k·d)** post-scoring |
| `structured` | SQL `WHERE` + topic-string post-filter | **O(N_filtered)** |
| `hybrid` | parallel semantic + structured, score-merged | **2× semantic** + O(k) merge |

Hybrid: semantic kept at `1/(1+distance)`, structured-only hits enter at `0.6 × score=1.0` [`code: src/store.ts:367-380`]. Fixed discount, not learned.

### 2.5 Dedup-on-write

[`code: src/store.ts:164-233`]. Cosine thresholds: **>0.92 = skip+touch, 0.75–0.92 = supersede chain, <0.75 = create new**. Supersede creates a new row with `supersedes = oldId`, marks `oldId.status='superseded', superseded_by=newId` [`code: src/store.ts:188-223, 763-769`] — old row preserved, full audit trail.

**Hidden cost flagged in v1, re-emphasized**: dedup bypasses ANN index. `findCandidates` SQL-filters to (type, scope, status='active') then `LIMIT 100`, then JS cosine over the candidate set [`code: src/store.ts:730-738`]. Below 100 active rows per (type, scope): correct. Above 100: silent candidate dropping, false-negative dedup, fragmenting store. This is the v1 finding that §1's MIT-12 framing now grounds in theory: SC chose a κ_max bound; the bound is well-formed but the *selection* policy under it is naive.

### 2.6 Lifecycle

`memory_compact` [`code: src/lifecycle.ts via index.ts:246-258`]: expire (`context`, `blocker` have TTLs), promote on access, demote on staleness, purge old superseded/archived. `access_count` bumped on retrieval, *not* on `loadSession` — explicit note at `src/store.ts:493-495` that bulk-bump creates self-reinforcing stale-row promotion.

### 2.7 Session load

[`code: src/store.ts:425-498`]. Tiered up to 38 rows: session-log (2) → critical non-task (15) → high-priority scoped (10) → tasks (8) → other context (3). Unique to SC + OpenClaw among the comparison set.

### 2.8 Transport, tools, concurrency, bridge

- HTTP on `127.0.0.1:18800/mcp` (Hono) + stdio for Cursor [`code: src/index.ts:324-326`, settings.json].
- 10 MCP tools: `memory_save, _search, _update, _delete, _load_session, _stats, _health, _compact, _task_list, _task_update`.
- Async-serialized `touchQueue` prevents lost-update races [`code: src/store.ts:127, 848-855`].
- `pmm-bridge.ts` exists — prior art for cross-system memory; relevant for Codex-memory design later.

A system with no transform layer might still produce great generalizations because the LLM caller does the distillation in prompt; a system with a transform layer might be doing it badly. Reader caveat: every property in this taxonomy is a proxy; consumers of this matrix should treat property scores as evidence to update on, not as conclusions.

## §6 Threat model applied to SuperClaude — MIT-12 protocol shape

Per attack surface from the security survey, applied to SC's actual deployment.

### 6.1 Write-time integrity (poisoning)

**SOURCE FACT** [`mit12: book-03-chunk-0495 pp.1479-1482 + book-03-chunk-0496 pp.1482-1485`]: *Machine Learning Systems* on data poisoning — "poisoning attacks the learning process itself, making them harder to detect and defend against." Online learning systems "are particularly susceptible to gradual poisoning. An attacker may introduce malicious samples incrementally, causing slow but steady shifts in model behavior."

**INFERENCE**: A memory system that learns from its own outputs (which is exactly what SuperClaude is — every `memory_save` becomes future context for the agent making future decisions) is structurally an online learning system. Gradual poisoning is therefore the dominant adversarial concern, not single-shot adversarial examples. Detection is hard because individual writes are plausibly legitimate.

**JUDGMENT for SuperClaude**: The current architecture's write-time integrity = `(Zod type+length validation) + (no semantic content validation) + (no authentication on the HTTP listener)` [`code: src/index.ts:55-67, settings.json mcpServers config`]. This is acceptable for a single-user developer box where the user is the only caller. The moment a second process gains localhost access (compromised dependency, malicious local script, second agent), gradual poisoning is feasible and undetectable. Mitigation: HTTP auth (O7 in §9) is the lowest-cost defense; semantic content classifiers are not warranted at current scale.

**OPEN QUESTION**: What is the actual frequency of localhost compromise in Hammad's environment? MIT-12 can't say; the threat model assumes worst-case which may be overconservative.

### 6.2 Retrieve-time integrity (injection)

**SOURCE FACT** [`mit12: book-03-chunk-0453 pp.1358-1361`]: White-box / grey-box / black-box adversarial knowledge spectrum. Adversaries with model access ≠ adversaries with input-only access; defenses differ.

**INFERENCE**: For SC, the threat-model question is: does an adversary need to plant rows (write-time) to influence retrieval, or can they craft queries that surface specific rows? Both. A row planted via §6.1 will be retrieved when its content semantically matches a future query — that's the design. The retrieve-time attack reduces to: plant a row whose vector lies near the vectors of expected future queries.

**JUDGMENT for SuperClaude**: SQL-injection on filters is closed [`code: src/store.ts:24-34, escapeStr + SAFE_STRING_REGEX`]. Query-semantic injection is open — but it requires §6.1 to succeed first. So §6.1 mitigation transitively closes most of §6.2.

**OPEN QUESTION**: Can an adversary influence the embedding model itself (BGE-M3 via `@openclaw/local-embeddings`)? MIT-12 has no answer — that's an upstream-supply-chain question.

### 6.3 Confidentiality

**SOURCE FACT** [`mit12: book-12-chunk-0222 pp.802-807`]: *Probabilistic ML Advanced Topics* — adversarial defenses based on robust optimization. Note the min-max formulation requires "solving both the non-concave inner maximization and the non-convex outer minimization problems" — generally NP-hard. Defensive certainty is computationally bounded.

**INFERENCE**: Confidentiality cannot be proven via local analysis; it must be designed structurally. For a memory system, the structural confidentiality boundary is the *visibility field's enforcement*. If `visibility: 'shared'` is a column but no code path checks it, the boundary doesn't exist.

**JUDGMENT for SuperClaude**: `visibility: 'private' | 'shared'` exists in the schema [`code: src/schema.ts:148`] but a grep of `src/store.ts` returns no enforcement. Today: single-user, irrelevant. The moment Codex memory + ACS are added (and per the requirements they will be), this becomes a P0 — without visibility enforcement, agent A can read agent B's private memories. Mitigation: O1 (enforce visibility in `memory_search`) is gating, not optional.

**OPEN QUESTION**: What constitutes "private" between cooperating agents that share a user? MIT-12 lens fairness has the construct-validity discipline (§3 footer) but the *policy* question is product-design, not theory.

### 6.4 Availability

**JUDGMENT**: LanceDB single-machine, no replication. Total loss possible on disk corruption. MIT-12 doesn't have much to say about backup engineering — that's operations, not theory. Standard remedy: external backup workflow, not in scope here.

### 6.5 Store/forget violations

**SOURCE FACT** [`mit12: book-10-chunk-0080 pp.238-241`]: *Fairness and Machine Learning* on the practical impossibility of clean retention/deletion guarantees in systems that learn from data. Once data has influenced a model, removing the raw data does not remove its influence.

**INFERENCE**: For a memory system, `memory_delete` that only archives [`code: src/store.ts:419-421`] is honest about this — the row's content is preserved but hidden. Hard-purge happens at compact time for old `superseded`/`archived` rows. The honest position is: there is no "delete" in any deep sense, only "hide and eventually purge".

**JUDGMENT for SuperClaude**: Adding a hard `memory_purge({id})` (O8 in §9) closes the GDPR-on-demand gap at the cost of losing audit trail for the purged row. The trade-off is acceptable: for content the user really wants gone, audit-trail loss is the feature.

**OPEN QUESTION**: How to handle "delete every memory derived from this conversation"? Requires conversation-graph tracking that SC doesn't have. Out of scope here.

### 6.6 Benign-persistence (stale fact returned as current)

**SOURCE FACT** [`mit12: book-03-chunk-0520 pp.1559-1561`]: "Safety in machine learning refers to the assurance that models behave predictably under normal conditions and fail in controlled, non-catastrophic ways under stress or uncertainty." Distribution shift is named as a primary safety risk — when training-time facts no longer reflect deployment-time reality.

**INFERENCE**: SC has TTLs on `context` and `blocker` only [`code: src/store.ts:801-807`]. Decisions, facts, lessons, preferences are *immortal* by default. A `decision` made in 2024 is returned with equal authority in 2026. This is structural benign-persistence and matches the survey's distribution-shift framing exactly.

**JUDGMENT for SuperClaude**: O2 (add `valid_from` / `valid_to`) closes this. ## §11 ACS multi-agent design principles (input to sub-project C)

> Full multi_agent lens. Principles, not designs. The ACS itself is current sub-project D and is not designed here.

### Principle 1: Decide centralization deliberately

**SOURCE FACT** [`mit12: book-08-chunk-0072 pp.249-252`]: *Multi-Agent RL* distinguishes **central learning** (single policy over joint-action space, using joint observation as input) from **decentralized training and execution** (each agent local, treats others as environment dynamics). Central has 3 problems: (1) joint reward must be transformed into single reward, hard in general-sum games, (2) joint-action space grows exponentially with N agents, (3) physical or virtual distribution may prevent communication. Decentralized has 3 downsides: (1) can't leverage info about others, (2) non-stationarity from concurrent training, (3) can't distinguish env stochasticity from other agents' actions.

**INFERENCE**: For Claude ↔ Codex ACS, the design must consciously pick a point on the central-vs-decentralized axis. Both agents writing to one shared SC-memory = central; each agent owning its own memory + communicating via a channel = decentralized. The two have different failure modes and different cost surfaces. Picking implicitly (i.e., "let's just have them call each other's APIs") usually produces the worst of both.

**JUDGMENT for ACS**: Decentralized — each agent owns its memory, ACS is the communication channel — because (a) physical distribution is real (different processes, possibly different hosts), (b) joint reward is impossible to specify cleanly (Claude and Codex have different goals at different moments), and (c) you want the architecture to scale to N=3, 4 agents without exponential blowup.

**OPEN QUESTION**: For sub-tasks where the agents *do* share a reward (e.g., "review this code for security"), should a temporary centralized memory layer be created? MIT-12 lens fairness reminds us that *shared reward across agents* is one of the rare clean cases for centralized training; pure decentralization may leave value on the table for these tasks.

### Principle 2: Specify reward structure before designing the protocol

**SOURCE FACT** [`mit12: book-08-chunk-0008 pp.39-42`]: MARL identifies three reward structures: **zero-sum** (one agent's reward is negative of the other's), **common reward** (all agents share a single reward — cooperative), **general-sum reward** (mixed-motive — agents have overlapping but not identical goals). The third is "among the most challenging tasks in MARL."

**INFERENCE**: Claude ↔ Codex is general-sum. Neither agent is purely cooperative (each has its own session, its own user-facing goals) nor purely adversarial. A "2nd opinion" interaction is cooperative on the *user's* welfare but competitive on *which agent's recommendation is adopted*. A "red-team" interaction is partially-adversarial-by-design — the red-teamer's reward is finding flaws, the blue-teamer's reward is shipping working code.

**JUDGMENT for ACS**: Name the reward structure per interaction type. Don't design a generic "channel" — design a per-protocol contract:
- **2nd opinion protocol**: common reward (user welfare); attribution preserved (who said what)
- **Red-team protocol**: zero-sum on "finds vulnerability vs. ships clean"; common-sum on user welfare (both want secure code shipped)
- **Hand-off protocol**: common reward, sequential, requires state-transfer semantics

**OPEN QUESTION**: For mixed-motive interactions (e.g., Claude proposes X, Codex challenges with Y, then both must agree on Z), how to specify the reward? MIT-12 multi-agent lens has equilibrium concepts (Nash, correlated, Stackelberg) but choosing one is a product-design call.

### Principle 3: Centralized value functions exist, but at exponential cost

**SOURCE FACT** [`mit12: book-08-chunk-0079 pp.271-274`]: Centralized value functions can address non-stationarity, partial observability, credit assignment, equilibrium selection — but are difficult to learn because the joint-action space grows exponentially. **Value decomposition** (coordination graph): when not all agents interact with each other, joint Q can be approximated as a sum over local interactions, exploiting graph sparsity.

**INFERENCE**: For ACS-level "what should Claude+Codex jointly do next?" reasoning, fully-centralized planning is intractable. But if interactions are sparse (most of the time, Claude and Codex operate independently; coordination only happens at handoff or review boundaries), the coordination graph is sparse, and local value decomposition is feasible.

**JUDGMENT for ACS**: Build for sparse coordination by default. The ACS should not assume both agents reason about each other's full state — only the slice exposed via shared messages. This matches the cost reality of the integration and matches book-08's coordination-graph optimization.

**OPEN QUESTION**: What is the right coordination-graph topology for Claude ↔ Codex? Probably a star (with the user at center) most of the time, occasionally a direct edge for hand-offs. MIT-12 can't say — this is design-space exploration.

### Principle 4: Free vs general communication is a hard taxonomy

**SOURCE FACT** [`mit12: book-04-chunk-0136 pp.570-573`]: Dec-POMDP subclasses are categorized by **observability** (degree to which shared state is observable) and **communication** — *free* (agents can share all observations, e.g., high-speed wireless) vs *general* (agents communicate imperfectly via their actions). "Free communication happens outside the model."

**INFERENCE**: ACS sits on this axis. A "free" channel = JSON-RPC over local socket with structured messages = agents have first-class shared observations. A "general" channel = agents communicate by acting in the shared world (e.g., both writing to SC memory and seeing each other's writes) = communication-via-action.

**JUDGMENT for ACS**: Start with free communication (explicit message channel, A2A protocol). Reasons: (a) cheaper to implement, (b) easier to reason about, (c) easier to red-team because messages are inspectable. The "communication via shared memory writes" pattern is more powerful but harder to verify — defer until free-channel limitations show up.

**OPEN QUESTION**: For sub-protocols (red-team, 2nd opinion), should they use free or general communication? Red-team in particular benefits from explicit message structure (request, response, finding). Hand-offs can use shared-memory writes when the state to transfer is too large for a single message.

### Principle 5: Agents must dimensionalize their interaction

**SOURCE FACT** [`mit12: book-08-chunk-0007 pp.36-39`]: MARL design questions are organized along: **Size** (N agents), **Knowledge** (what agents know about each other), **Observability** (what can be observed), **Rewards** (zero-sum / common / general), **Objective** (equilibrium type), **Centralization & Communication**.

**INFERENCE**: Before any ACS protocol is implemented, these 6 dimensions must be filled in *for the specific Claude ↔ Codex deployment*. Without that, design is premature.

**JUDGMENT for ACS scoping**: When sub-project D is opened, the first deliverable should be a one-pager filling these 6 dimensions explicitly. That document gates the protocol design.

**OPEN QUESTION**: Do Claude and Codex know each other's *capabilities*? Per A2A's Agent Cards, capability advertisement is part of the protocol. Both should advertise; both should validate.

### Principle 6: Equilibria are real but not always reachable

**SOURCE FACT** [`mit12: book-09-chunk-0022 pp.96-100`]: Every game has at least one Nash equilibrium, but humans (and agents) "often do not play a Nash equilibrium strategy" — because of cognitive limitations or doubt that the opponent can compute it. Behavioral game theory uses logit-level-k models capturing bounded rationality.

**INFERENCE**: For ACS interactions, do not assume agents will converge on equilibrium behavior. Both Claude and Codex are bounded-rationality agents — they will misjudge each other. The protocol must tolerate that misjudgment without producing wrong answers.

**JUDGMENT for ACS**: Build *human-in-the-loop escalation* paths for disagreements. When agents disagree about "is this a security finding?" the protocol should surface the disagreement to the user with both agents' reasoning, rather than forcing a winner. This is the multi-agent equivalent of the §9 reversibility/irreversibility-gate discipline.

**OPEN QUESTION**: At what level of disagreement should escalation fire? Calibration question; needs user-preference input.

### Summary: 6 principles, no design

The above are *constraints* the ACS design must satisfy, not the design itself. When sub-project D opens, the design phase starts from this constraint set, fills in the Principle-5 dimensions, and produces a concrete protocol spec.

**INFERENCE**: Without `agent_id`, every cross-agent memory operation requires string-parsing of `scope` to recover identity. Brittle. And the recovery is *lossy* — once written, you can't tell whether `scope: 'project:openclaw'` was written by Claude, Codex, or a human directly via curl.
**JUDGMENT**: Add `agent_id` to schema before any cross-agent work. Backfill existing rows to `agent:claude`. This is cheap (1 dev-day) and unblocks everything multi-agent.
**OPEN QUESTION**: What identifier scheme? Free string? Enum? Public-key fingerprint? Depends on whether the ACS will eventually use signed messages (in which case PK identity is right) or just trusted-process identity (in which case enum is fine).

### B.5 — Why O10 (MaTTS-Parallel) is the highest-value-per-dollar memory upgrade if a benchmark threshold matters

**SOURCE FACT 1** [`paper: arXiv:2509.25140 abstract`]: ReasoningBank + MaTTS = +34.2% effectiveness, -16% interaction steps on WebArena-Admin.
**SOURCE FACT 2** [`mit12: book-06 on credit assignment + book-07 on distributional value`]: parallel rollouts with contrast = textbook variance-reduction + bias-reduction; the +34.2% is plausible because the underlying mechanism is well-understood, not novel.
**SOURCE FACT 3** [`mit12: book-09-chunk-0022 pp.96-100`]: bounded rationality — a single rollout is a single sample from a high-variance distribution; multiple rollouts reduce that variance.
**INFERENCE**: MaTTS is the closest thing in current literature to a "free lunch" for agent memory quality — if you're willing to spend 3× compute per session, you get measurably better memory induction. The cost is real (3× tokens) but the benefit is grounded in well-understood theory.
**JUDGMENT**: O10 is HIGH-value but requires orchestrator-level changes, not memory-level. It belongs in the SuperClaude-orchestrator roadmap, not the SC-memory MCP. Mention it here so it isn't lost.
**OPEN QUESTION**: At what task complexity threshold does MaTTS pay off? ReasoningBank's WebArena-Admin is complex; for simple lookups, 3× compute for marginal gain is a bad trade.
